## Blocking Gap

### Preferences Save Flash Can Use Old Locale After Locale Change

**Affected requirements:** PREF-03, TRN-04, VERI18N-02

`PreferencesController#create` and `PreferencesController#update` translate the save notice before redirect:

```ruby
flash[:notice] = t('preferences.saved')
redirect_to preferences_path
```

When the request changes `preference.locale` from `ja` to `en`, the current request still runs under the old locale. The redirected GET then resolves `en`, but the flash notice is already a Japanese string. This creates a mixed-language preferences page immediately after changing language.

**Expected:** After saving `locale='en'`, the redirected `/preferences` page renders both page chrome and `preferences.saved` notice in English.

**Actual:** Page chrome resolves to English, but the save notice can remain Japanese because it was translated before the redirect.

## Closed Prior Gap

### OTP Challenge Now Uses Saved Locale

Phase 18.1 closed the previous blocking gap:

- `Localization#saved_locale` now delegates to `saved_locale_user&.preference&.locale`.
- `saved_locale_user` returns `current_user` when signed in, or `User.find_by(id: session[:otp_user_id])` while pending OTP.
- `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` still guards every locale candidate before `I18n.with_locale`.
- `test_保存済みlocaleがenのユーザはOTPページを英語で表示する` proves saved English OTP rendering without `Accept-Language`.
- `test_保存済みlocaleが未対応値のOTPページはデフォルト日本語にフォールバックする` proves stale unsupported saved locale fallback.
