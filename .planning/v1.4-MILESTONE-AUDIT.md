## Blocking Gap

### OTP Challenge Does Not Use Saved Locale

**Affected requirements:** I18N-02, PREF-02, AUTHI18N-02, VERI18N-02

`Localization#saved_locale` only reads `current_user.preference.locale` when `user_signed_in?` is true:

```ruby
def saved_locale
  return nil unless user_signed_in?
  current_user.preference&.locale
end
```

During 2FA sign-in, `Users::SessionsController#create` stores `session[:otp_user_id]` and redirects to `Users::TwoFactorAuthenticationController#show`. At that point the user has passed password authentication but is not yet signed in, so `saved_locale` returns nil. The OTP page therefore uses `Accept-Language` or the default `:ja`, not the saved account preference.

**Expected:** A user with saved `locale='en'` and 2FA enabled sees the OTP challenge in English after password authentication, even without an `Accept-Language: en` header.

**Actual:** The OTP challenge falls back to Japanese unless the request also carries `Accept-Language: en`.
