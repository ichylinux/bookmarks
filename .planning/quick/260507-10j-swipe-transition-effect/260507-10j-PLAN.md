  <action>
Replace the mobile `display: none / display: block` column-switching mechanism with a CSS custom-property-driven horizontal slide.

Inside the existing `@media (max-width: $portal-mobile-breakpoint - 1px)` block, replace:

```scss
.portal .gadgets.portal-column {
  display: none;
  width: 100%;
  float: none;
  padding-bottom: 0px;
}

@for $i from 0 through 7 {
  .portal.portal--column-active-#{$i} #column_#{$i} {
    display: block;
  }
}
```

With a sliding track layout:

```scss
// Outer clipping container — one column-width wide, no horizontal overflow
.portal {
  overflow: hidden;
  position: relative;
}

// Inner horizontal track — all columns laid out side-by-side
.portal-track {
  display: flex;
  flex-direction: row;
  transition: transform 280ms cubic-bezier(0.35, 0, 0.25, 1);
  will-change: transform;
  transform: translateX(calc(-100% * var(--portal-active-index, 0)));
}

.portal .gadgets.portal-column {
  flex: 0 0 100%;
  width: 100%;
  min-width: 0;
  float: none;
  padding-bottom: 0;
}
```

The `@for $i from 0 through 7` block and the old `display: none/block` rules must be removed — the sliding track replaces them entirely.

IMPORTANT: The `.portal-track` wrapper div does not yet exist in the ERB template — Task 2 adds it to the JS so that the HTML is updated without touching ERB. Alternatively, wrap in the ERB partial. Use the ERB approach: in `app/views/welcome/_portal_column_section.html.erb`, wrap the `portal-column` divs inside a `<div class="portal-track">` directly inside `.portal`. This is the simpler path — no JS DOM manipulation needed.

Update `_portal_column_section.html.erb` to add the wrapper:

<task type="auto">
  <name>Task 2: Drive slide via CSS custom property in portal_mobile_tabs.js</name>
  <files>app/assets/javascripts/portal_mobile_tabs.js</files>
  <action>
Update `activateColumn` to set the `--portal-active-index` CSS custom property on the `.portal` element so the CSS transition fires on every column change (tab click and swipe).

Current `activateColumn`:
```javascript
const activateColumn = function($portal, $tabs, index) {
  $tabs.find('.portal-column-tab').each(function() { ... });
  syncPortalClasses($portal, index);
  if (isMobileViewport()) {
    window.localStorage.setItem(STORAGE_KEY, String(index));
  }
};
```

Add one line after `syncPortalClasses`:
```javascript
$portal[0].style.setProperty('--portal-active-index', index);
```

This pairs with the CSS `transform: translateX(calc(-100% * var(--portal-active-index, 0)))` on `.portal-track`.

The `--portal-active-index` default is `0` in CSS, so on first render before JS runs the first column is visible with no flash.

No other changes to the swipe gesture detection logic — the existing `touchstart / touchmove / touchend` handlers already call `activateColumn` on gesture completion. The CSS transition provides the animation automatically.

Optionally, add a live drag preview: during `touchmove`, update `--portal-active-index` to a fractional value (e.g. `currentIndex - dx/containerWidth`) so the track follows the finger in real time, and on `touchend` snap to the target integer index (which triggers the CSS transition for the last portion of the movement). This is OPTIONAL — implement if it fits cleanly; skip if it adds risk. The core requirement is the transition on column change.

Keep all existing contract assertions in `test/assets/portal_mobile_tabs_js_contract_test.rb` passing — do not rename or remove any of the strings those tests assert on.
