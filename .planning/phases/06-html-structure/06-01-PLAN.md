<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser → Rails | HTTP request renders ERB layout; only authenticated requests can reach drawer markup |
| Authenticated session → Logout endpoint | DELETE /users/sign_out via Rails UJS form spoofing on the drawer's logout link |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-06-01 | Information Disclosure | Drawer nav links rendered in HTML response | mitigate | Wrap drawer + overlay markup in `<% if user_signed_in? %> ... <% end %>` block (matches D-04 and existing `render 'common/menu' if user_signed_in?` pattern). Verified by `test_非ログイン時はドロワーが存在しない` integration test — guest GET / redirects without rendering drawer. |
| T-06-02 | Tampering | Logout link HTTP method downgrade (GET instead of DELETE) | mitigate | Use `link_to 'ログアウト', destroy_user_session_path, method: 'delete'` verbatim (Rails UJS emits `data-method="delete"` for form-based DELETE). Verified by `assert_select '.drawer a[href=?][data-method=?]', destroy_user_session_path, 'delete'`. |
| T-06-03 | Tampering (XSS) | Hamburger `aria-label` and nav link labels | accept | Labels are static literals in ERB source, not user input. ERB auto-escapes any inadvertent interpolation. No mitigation beyond standard Rails escaping required. |
| T-06-04 | Spoofing | CSRF on logout DELETE | mitigate | `csrf_meta_tags` already in `<head>`; Rails UJS injects authenticity token into the spoofed form. No additional Phase 6 work — pre-existing baseline. |
| T-06-05 | Elevation of Privilege | Drawer renders for users not yet authenticated (race) | accept | Standard Devise `user_signed_in?` semantics — same guard already used for `render 'common/menu'`. Risk parity with existing dropdown; no new exposure. |
</threat_model>
