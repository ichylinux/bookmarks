# Phase 18.2 Context — Preferences Locale Change Flash

## Audit Gap

The v1.4 milestone audit found one remaining blocking integration gap after Phase 18.1 closed the pending OTP saved-locale issue.

When a user changes their language preference, `PreferencesController#create` and `PreferencesController#update` store the save notice as an already-translated string before redirect:

```ruby
flash[:notice] = t('preferences.saved')
redirect_to preferences_path
```

The current request is still running under the old locale because `Localization#set_locale` resolved `I18n.locale` before the preference update. The redirected GET then uses the newly saved locale for the page chrome, but the flash string was already materialized in the old locale. This can produce a mixed-locale preferences page immediately after changing language.

## Affected Requirements

- `PREF-03`: User sees the preferences page itself translated after changing language.
- `TRN-04`: User sees flash messages, controller alerts, and validation-facing labels in the active locale.
- `VERI18N-02`: User-facing tests cover representative Japanese and English UI paths for layout, preferences, core gadgets, and auth/2FA surfaces.

## Expected Behavior

- Starting from saved `locale = 'ja'`, submitting preferences with `locale = 'en'` redirects to `/preferences`.
- The redirected page renders `<html lang="en">`.
- The redirected page shows `.flash-notice` as `I18n.t('preferences.saved', locale: :en)`.
- The reverse direction (`en` to `ja`) behaves symmetrically.
- Non-locale preference saves keep existing behavior.

## Likely Implementation Shape

Prefer keeping the flash as a string, but translate it after the new preference has been saved and under the new effective locale.

Possible local pattern:

```ruby
@user.transaction do
  @user.save!
end

flash[:notice] = I18n.with_locale(saved_notice_locale) { t('preferences.saved') }
redirect_to preferences_path
```

The locale helper should derive from the saved preference when it is supported, otherwise preserve the normal fallback behavior. Any candidate locale must continue to pass `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` before use.

## Regression Tests

Target file: `test/controllers/preferences_controller_test.rb`.

Required coverage:

1. `ja -> en` locale update:
   - start with `user.preference.locale = 'ja'`
   - sign in
   - PATCH preferences with `locale: 'en'`
   - follow redirect
   - assert path `/preferences`
   - assert `html[lang=en]`
   - assert `.flash-notice` equals English `preferences.saved`
2. `en -> ja` locale update:
   - same path in reverse
   - assert `html[lang=ja]`
   - assert `.flash-notice` equals Japanese `preferences.saved`
