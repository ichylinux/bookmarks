<objective>
Close the v1.4 milestone audit gap `preferences-locale-change-flash-old-locale`. When a user
changes their language preference, `PreferencesController#create` and `#update` currently
materialize `flash[:notice] = t('preferences.saved')` while the request is still running
under the OLD locale (because `Localization#set_locale` resolved I18n.locale before the save).
The redirected GET correctly renders new-locale page chrome, but the flash string is in the
old locale, producing a mixed-locale page.

Fix: after `@user.save!`, derive the saved candidate locale from `@user.preference.locale`,
validate against `Preference::SUPPORTED_LOCALES`, and translate the flash under that locale
via `I18n.with_locale(candidate) { t('preferences.saved') }`. If candidate is missing or
unsupported, fall back to plain `t('preferences.saved')` (current behavior).

Output:
- `app/controllers/preferences_controller.rb` updated in both `#create` and `#update`
- `test/controllers/preferences_controller_test.rb` extended with two regression tests covering ja->en and en->ja directions
- Full verification gate green per CLAUDE.md
</objective>

  ```ruby
  def saved_notice
    candidate = @user.preference&.locale
    if candidate.present? && Preference::SUPPORTED_LOCALES.include?(candidate.to_s)
      I18n.with_locale(candidate.to_sym) { t('preferences.saved') }
    else
      t('preferences.saved')
    end
  end
  ```

  Constraints:
  - Do NOT modify `Localization` concern.
  - Do NOT bypass `around_action :set_locale`; this helper only translates the flash string,
    it does not change the request's `I18n.locale`.
  - Do NOT remove or reorder the existing `attrs[:preference_attributes][:locale].presence`
    normalization or the `@user.transaction { @user.save! }` block.
  - Helper MUST use `@user.preference&.locale` (the just-saved value), NOT `params[...][:locale]`,
    so a non-locale-touching save (e.g., font_size only) reads the existing persisted locale.
  - Helper MUST guard with `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` so a
    nil/empty/stale-unsupported candidate falls back to plain `t('preferences.saved')`
    (which equals the request's `I18n.locale`, i.e., current behavior).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser form -> PreferencesController | User-submitted `preference_attributes[:locale]` arrives as a string |
| Stored Preference -> I18n.with_locale | Saved locale value is read back from `@user.preference.locale` after save |
| Controller flash -> rendered layout | `flash[:notice]` is rendered into `.flash-notice` on the redirected GET |

## STRIDE Threat Register

| Threat ID | Category | Component | Severity | Disposition | Mitigation Plan |
|-----------|----------|-----------|----------|-------------|-----------------|
| T-18.2-01 | T — Tampering / Injection | Locale candidate fed to `I18n.with_locale` | Low | mitigate | Explicit whitelist gate `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` before `I18n.with_locale`; only `%w[ja en]` reach I18n. No I18n key injection or locale-spoofing path opens. |
| T-18.2-02 | T — Tampering | `preference_attributes[:locale]` permitted param | Low | mitigate | Existing model-level `validates :locale, inclusion: { in: SUPPORTED_LOCALES }, allow_nil: true` rejects unsupported values at `save!`; helper reads back the persisted (validated) value, not the raw param. |
| T-18.2-03 | I — Information disclosure | Flash text in old-vs-new locale | None | accept | Flash content is the static key `preferences.saved` (no PII); only the rendering language changes. |
| T-18.2-04 | E — Elevation of privilege | Controller change scope | None | accept | Change is confined to a flash-translation helper; does not alter authentication, authorization, around_action ordering, or session handling. |
| T-18.2-05 | D — Denial of service | I18n.with_locale call | None | accept | Single synchronous translation lookup; no I/O, no unbounded loop. |
</threat_model>
