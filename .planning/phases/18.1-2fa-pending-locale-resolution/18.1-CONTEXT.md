# Phase 18.1 Context — 2FA Pending Locale Resolution

## Why This Phase Exists

The v1.4 milestone audit found one blocking cross-phase integration gap:

> A user with a saved locale preference does not keep that locale on the 2FA OTP challenge page while they are between password authentication and OTP completion.

Phase 14 verified the normal locale resolution pipeline:

1. saved account locale
2. valid `Accept-Language` match
3. default Japanese

Phase 18 verified OTP rendering in Japanese by default and English via `Accept-Language`. The missing integration path is a saved English user with 2FA enabled and no `Accept-Language: en` header.

## Root Cause

`Localization#saved_locale` currently only reads `current_user.preference.locale` when `user_signed_in?` is true.

During 2FA sign-in:

1. `Users::SessionsController#create` validates email/password.
2. For a 2FA-enabled user, it stores `session[:otp_user_id]`.
3. It redirects to `users_two_factor_authentication_path`.
4. The user is not yet signed in, so `user_signed_in?` is false.
5. `Localization#saved_locale` returns nil.
6. The OTP page uses `Accept-Language` or falls back to `:ja`.

## Requirements Reopened

- `I18N-02` — saved account locale must participate in locale resolution.
- `AUTHI18N-02` — custom 2FA authentication pages must render in Japanese or English.
- `VERI18N-02` — representative auth/2FA locale tests must include the saved-locale OTP path.

## Expected Behavior

A user with:

- `preference.locale = 'en'`
- two-factor authentication enabled
- no `Accept-Language: en` header

should see the OTP challenge page rendered with:

- `<html lang="en">`
- `Verification code` label
- English 2FA page copy

## Likely Implementation Shape

Keep the Phase 14 safety invariants:

- `I18n.with_locale` remains in an `around_action`.
- `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` must still whitelist every candidate before `with_locale`.
- Unsupported/stale stored locales must silently fall through to `Accept-Language` or `:ja`.

Likely code change:

- Extend `Localization#saved_locale` so that, when no user is signed in, it checks a pending 2FA user from `session[:otp_user_id]` and returns that user's `preference.locale`.
- Avoid signing in the user early.
- Avoid trusting arbitrary params or headers.

## Required Regression Test

Add a test in `test/controllers/two_factor_authentication_controller_test.rb`:

1. Set `user.preference.update!(locale: 'en')`.
2. Enable 2FA.
3. POST `user_session_path` with correct password and no `Accept-Language` header.
4. GET `users_two_factor_authentication_path` with no `Accept-Language` header.
5. Assert success.
6. Assert `html[lang=en]`.
7. Assert label text `I18n.t('two_factor.code_label', locale: :en)`.

Also preserve existing default Japanese and explicit Accept-Language tests.
