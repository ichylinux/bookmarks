config.i18n.available_locales = %i[ja en] # mirrors Preference::SUPPORTED_LOCALES

**Critical:** Use the inline literal `%i[ja en]`, NOT `Preference::SUPPORTED_LOCALES.map(&:to_sym)`. Zeitwerk has not loaded application models when config/application.rb executes. Referencing `Preference::SUPPORTED_LOCALES` here causes `uninitialized constant Preference` at boot time.

I18n.config.enforce_available_locales = true

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| DB write → model | Locale value written to preferences table must pass model validation before persisting |
| model → I18n config | SUPPORTED_LOCALES drives both validation and (via parallel literal) boot-time available_locales |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Tampering | Preference#locale DB column | mitigate | `validates :locale, inclusion: { in: SUPPORTED_LOCALES }, allow_nil: true` rejects invalid locale values at write time; unsupported locales cannot be persisted via ActiveRecord |
| T-14-02 | Tampering | I18n.available_locales boot config | mitigate | `config.i18n.available_locales = %i[ja en]` combined with `enforce_available_locales = true` prevents any unsupported locale from being set on the I18n system; raises `I18n::InvalidLocale` on any attempt |
| T-14-03 | Elevation of Privilege | Direct SQL / stale DB value bypassing model validation | accept | Personal app with single authenticated owner; DB-level CHECK constraint not warranted. Plan 02's pipeline whitelist provides second read-time defense layer |
</threat_model>
