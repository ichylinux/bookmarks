Create `app/controllers/concerns/localization.rb` with the following exact content:

```ruby
module Localization
  extend ActiveSupport::Concern

  included do
    around_action :set_locale
  end

  private

  def set_locale(&block)
    I18n.with_locale(resolved_locale, &block)
  end

  def resolved_locale
    [saved_locale, accept_language_match].each do |candidate|
      return candidate.to_sym if candidate && Preference::SUPPORTED_LOCALES.include?(candidate.to_s)
    end
    I18n.default_locale
  end

  def saved_locale
    return nil unless user_signed_in?
    current_user.preference&.locale
  end

  def accept_language_match
    parse_accept_language(request.env['HTTP_ACCEPT_LANGUAGE'])
  end

  def parse_accept_language(header)
    return nil if header.blank?
    header
      .split(',')
      .map { |entry|
        tag, q = entry.strip.split(';q=')
        q_value = q ? q.to_f : 1.0
        [tag.to_s.strip[0, 2], q_value]
      }
      .sort_by { |_, q| -q }
      .find { |tag, _| Preference::SUPPORTED_LOCALES.include?(tag) }
      &.first
  rescue
    nil
  end
end
```

**Critical implementation notes:**

1. `around_action :set_locale` (NOT `before_action`) — `I18n.locale` is thread-local and is NOT reset between Puma requests. `I18n.with_locale` saves and restores the previous locale atomically, preventing locale bleed across requests on the same thread.

2. The whitelist check `Preference::SUPPORTED_LOCALES.include?(candidate.to_s)` in `resolved_locale` MUST run before `I18n.with_locale`. After Plan 01 restricts `I18n.available_locales` to `[:ja, :en]`, passing any other locale to `I18n.with_locale` raises `I18n::InvalidLocale`.

3. `parse_accept_language` uses q-value sorting to respect browser preference order (e.g., `ja-JP;q=0.9, en-US;q=0.8` correctly resolves to `ja`). The `rescue; nil` at the end handles blank, nil, or malformed headers without raising.

4. `saved_locale` returns nil (not an object) for signed-out users (`unless user_signed_in?`) and for users with no persisted preference (`preference&.locale`).

5. No `?locale=` param is read anywhere in this concern (per D-04 / Claude's Discretion: params approach deferred).

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP request → controller | HTTP_ACCEPT_LANGUAGE header is attacker-controlled; must be parsed safely |
| DB locale value → I18n | Stored locale value may be stale or from direct SQL; must be whitelisted before use |
| I18n.locale → HTML output | Resolved locale is interpolated into lang attribute; must be a safe symbol string |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-14-01 | Tampering | HTTP_ACCEPT_LANGUAGE header input | mitigate | `parse_accept_language` uses `rescue; nil` to neutralize malformed input; whitelist check `SUPPORTED_LOCALES.include?(tag)` rejects every non-supported tag before it reaches `I18n.with_locale` |
| T-14-02 | Tampering | Stale or direct-SQL locale value in preferences table | mitigate | `resolved_locale` checks `SUPPORTED_LOCALES.include?(candidate.to_s)` before returning any stored value; invalid stored locales silently fall through to Accept-Language then :ja |
| T-14-03 | Denial of Service | I18n::InvalidLocale exception triggered by unsupported locale | mitigate | Whitelist guard in `resolved_locale` runs before `I18n.with_locale`; `enforce_available_locales = true` will never be reached with an out-of-set locale |
| T-14-04 | Information Disclosure | lang attribute leaking internal locale state | accept | lang="ja" / lang="en" are public UI information; no sensitive data; WCAG/accessibility standard requires this attribute |
| T-14-05 | Spoofing | Thread-local I18n.locale from previous request persisting | mitigate | `around_action` with `I18n.with_locale` saves and restores locale atomically; previous-request locale cannot bleed into current request |
</threat_model>
