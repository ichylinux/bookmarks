# Phase 33.1 Research: TEST-02 Tab-Click Regression Contract (JS/Minitest)

## Objective

Define a robust, low-maintenance contract for mobile tab-click regression behavior in `portal_mobile_tabs.js` using Minitest-level static assertions, so refactors cannot silently break tab-driven column switching semantics.

## Sources Reviewed

- `.planning/ROADMAP.md` (Phase 33.1 goal and success criteria)
- `.planning/REQUIREMENTS.md` (`TEST-02` requirement context)
- `app/assets/javascripts/portal_mobile_tabs.js` (actual interaction flow)
- `test/assets/portal_mobile_tabs_js_contract_test.rb` (existing JS contract suite)
- `.planning/phases/032-1-close-gap-swipe-01-right-swipe-e2e-contract/032-1-01-PLAN.md` (recent gap-closure plan pattern)
- `.planning/phases/032-1-close-gap-swipe-01-right-swipe-e2e-contract/032-1-RESEARCH.md` (recent contract-gap rationale)

## Findings

### 1) Shared activation path already exists and is testable via source contract

`portal_mobile_tabs.js` currently routes both tab-click and swipe-end transitions into the same `activateColumn($portal, $tabs, index)` function. This is exactly the regression contract Phase 33.1 needs to lock down.

### 2) Existing contract tests are close but not explicit enough for tab-click semantics

`portal_mobile_tabs_js_contract_test.rb` already asserts shared flow in one test, but does not explicitly pin:

- click handler's index extraction (`data-portal-column-index`)
- click path's portal/tabs resolution (`closest` + `next`)
- active UI sync inside activation flow (`portal-column-tab--active`, `aria-selected`)
- class sync (`portal--column-active-N`)

These are the high-value regression points for "tab click drives mobile column switch behavior."

### 3) Minitest static contract is appropriate here

This phase is about contract explicitness, not browser interaction E2E. Static source assertions are fast, deterministic, and prevent accidental refactors from removing key interaction wiring.

## Recommended Implementation

1. Extend `test/assets/portal_mobile_tabs_js_contract_test.rb` with one focused contract test for tab-click semantics:
   - assert click handler parses tab index from `data-portal-column-index`
   - assert click handler resolves portal via tabs container
   - assert click handler invokes shared `activateColumn` path

2. Strengthen activation-flow contract assertions (either in existing test or a new dedicated test):
   - `portal-column-tab--active` toggle
   - `aria-selected` synchronization
   - `portal--column-active-N` class synchronization via `syncPortalClasses`

3. Keep persistence/restore and swipe contracts untouched to avoid scope creep.

## Risks and Guardrails

- Avoid changing production JS behavior in this phase unless tests reveal an actual defect.
- Keep assertions resilient to formatting changes (regex where suitable, `assert_includes` for stable literals).
- Ensure `TEST-02` is referenced directly in plan metadata for coverage gate compatibility.
